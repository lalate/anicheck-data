# DATA_PIPELINE_SPEC.md: アニちぇっく データ収集パイプライン仕様書

このドキュメントは、「アニちぇっく」エコシステムのデータ収集・管理リポジトリ (`anicheck-data`) のパイプライン仕様を定義します。

## 1. パイプラインの全体像と目的

このパイプラインの目的は、Grok AIとGitHub Actionsを連携させ、アニメの放送情報を **半自動** で収集・構造化し、Flutterアプリと静的Webサイトのデータソースとして提供することです。

パイプラインは、大きく分けて2つのフェーズで構成されます。

1.  **シーズン初期化フェーズ（手動実行）**: 新しいアニメクールが始まる際に、そのシーズンの全作品の基本データを生成する。
2.  **日次更新フェーズ（自動実行）**: シーズンが始まった後、日々の放送情報を自動で更新し続ける。

## 2. シーズン初期化フェーズ（手動）

このフェーズは、新しいクール（例: 2026年冬アニメ）の放送が始まる前後に、開発者が手動でスクリプトを実行します。

### Step 1: 作品リストのインデックス取得 (`grok_anime_fetcher.py`)

-   **目的**: 対象となるシーズンのアニメ作品リストをAIから取得し、今後の処理の「目次」を作成する。
-   **トリガー**: 開発者による手動実行 (`python3 grok_anime_fetcher.py`)。
-   **インプット**:
    -   スクリプト内で定義されたシーズンキーワード（例: "2026年冬アニメ"）。
-   **プロセス**:
    1.  Grok AIに対し、「{シーズンキーワード}の主要な深夜アニメをリストアップし、1から始まる通し番号をつけて列挙してください」というプロンプトを送信する (`fetch_anime_list_index` 関数)。
    2.  AIから返却された、通し番号付きの作品名リスト（テキスト形式）を変数に保持する。
-   **アウトプット**: 作品名の通し番号付きリスト（例: `1. 葬送のフリーレン
2. 薬屋のひとりごと...`）。

### Step 2: 詳細情報のバッチ取得 (`grok_anime_fetcher.py`)

-   **目的**: Step 1で作成した「目次」に基づき、AIの出力制限を回避しながら、全作品の詳細情報を安全に取得する。
-   **プロセス**:
    1.  Step 1で取得した作品リストを、10件ごとのバッチに分割する。
    2.  各バッチごとに、Grok AIに対し「リストの{開始番号}から{終了番号}までの作品について、詳細情報をJSON形式で出力してください」というプロンプトを繰り返し送信する (`fetch_anime_details_batch` 関数)。
-   **アウトプut**: 全バッチの実行結果を結合した、構造化されていない生テキストファイル (`raw_grok_output_batched.txt`)。

### Step 3: データの精製と構造化 (`anime_factory.py`)

-   **目的**: AIが生成した生テキストから、クリーンで構造化されたJSONデータを抽出・分類し、初期データセットを完成させる。
-   **トリガー**: 開発者による手動実行 (`python3 anime_factory.py`)。
-   **インプット**: `raw_grok_output_batched.txt` (ファイル名を `raw_grok_output.txt` に変更して使用)。
-   **プロセス**:
    1.  生テキストファイルから、Markdownのコードブロック（` ```json ... ``` `）に囲まれたJSON部分のみを全て抽出する。
    2.  抽出した各JSONオブジェクトのキー構造を分析し、「Master」「Episode」「Broadcast」の3種類に分類する。
-   **アウトプット**:
    -   `master.json`: 全作品のMasterデータをまとめたリスト。
    -   `episode.json`: 全作品のEpisodeデータをまとめたリスト。
    -   `broadcast.json`: 全作品のBroadcastデータをまとめたリスト。
    -   `watch_list.json`: 日次更新の監視対象リスト。

## 3. 日次更新フェーズ（自動）

このフェーズは、シーズンが始まると毎日自動で実行され、情報の鮮度を保ちます。

### Step 1: 日次更新の実行 (`anicheck_daily.py`)

-   **目的**: 放送日時の変更、サブタイトル、あらすじなど、日々変動する可能性のある情報を最新化する。
-   **トリガー**: GitHub Actions (`.github/workflows/daily_fetch.yml`) によるスケジュール実行 (毎日 AM 8:00 JST)。
-   **インプット**: `current/watch_list.json`。
-   **プロセス**:
    1.  `watch_list.json` を読み込み、監視対象の各アニメについてループ処理を行う。
    2.  各アニメに対し、Grok AIに「この作品の{次の話数}の最新情報を教えてください」という、ハルシネーションを厳しく抑制するプロンプトを送信する。
        -   **プロンプトの要点**:
            -   事実ベースの制約: 「提供された情報のみに基づき、推測するな」
            -   Chain of Thought: 「Step1: 分析, Step2: 検索, Step3: JSON構築」
            -   ユーモアの封印: 「純粋なJSONのみを出力し、解説や挨拶は含めるな」
    3.  AIから返却されたJSONデータをパースする。
-   **アウトプット**: `current/` ディレクトリ内に、作品ごとの個別JSONファイルを上書き保存する。
    -   `{anime_id}_master.json`
    -   `{anime_id}_episode.json`
    -   `{anime_id}_broadcast.json`
    -   更新された `watch_list.json` (話数をインクリメント)

## 4. データ品質管理

-   **目的**: コミュニティからの修正提案（Pull Request）の品質を担保する。
-   **トリガー**: GitHub Actions (`.github/workflows/validate_pr.yml`) による、`current/**/*.json` へのPull Request。
-   **プロセス**: `scripts/validate_json.py` を実行し、提案されたJSONファイルが定義されたスキーマに準拠しているか、必須フィールドが欠けていないかを自動検証する。
