はい、そのイメージでほぼ正しいです！

daily_schedule.json は、**「今日（実行日）に、どの放送局で、何時から、何話（どのエピソード）を放送/配信するのか」**という情報をまとめたファイルとして機能させるのが最適です。

つまり、アプリのホーム画面で最初に表示される「今日のアニメ放送予定表」のデータ源になるものです。

イメージを具体的に言うと

- ユーザーがアプリを開いたら「今日（2026-03-21）の放送予定」が一覧で見える
- 各行は「この局で、この時間から、この作品の第何話をやる」という情報
- 例:

- TOKYO MX で 00:00 から 「葬送のフリーレン」 第28話
- 毎日放送 で 01:25 から 「呪術廻戦 第3期」 第15話（遅延あり）

これを daily_schedule.json に持たせておくと、  
アプリ側で「時間順ソートして表示」「局でフィルタ」「もうすぐ放送の通知」などを実現しやすくなります。

持つべき情報（今日の放送に特化させるために）

- 過去の放送情報は不要（それは episodes/ フォルダや履歴ファイルで保持）

- daily_schedule は**「今日見るべきアニメ」**に特化させるのがアプリとして自然

あなたのアイデアを整理すると、以下のような運用になりそうで、これならかなり現実的でメンテナンスもしやすい形になると思います。

提案のまとめ（今のイメージを固めた形）

1. watch_list.json の役割を「監視対象作品の最新確認状態」に特化

- ep_num → 「Grokが最後に確認した最新話数」として扱う
- 毎日Grokに「{{アニメタイトル}}の最新話」を聞く
- Grokが返してきた ep_num をそのまま上書き保存 → こっちでインクリメントする手間がなくなる（Grokが最新値を教えてくれる）

3. 局ごとの進捗は完全に別管理

- daily_schedule.json や broadcast_history.json で「局ごと・日ごとの放送実績」を記録
- ユーザー側で「自分がどの局を見ているか」を設定（または複数選択）
- 次に見るべき話数は「ユーザーが見ている局の最終放送話数 + 1」

5. 情報収集プロンプトのピンポイント化（Grokに聞くときの理想形）

作品名：{{アニメタイトル}}

対象：この作品の**最新放送済み話**（または現在最も新しいエピソード）

出力形式：

{

  "latest_ep_num": 整数（最新話数）、

  "latest_sub_title": "サブタイトル" または null,

  "latest_broadcast_date": "YYYY-MM-DD" または null,

  "source": "公式サイト" または "ANN" など具体的なソース名,

  "updated_at": "YYYY-MM-DDTHH:MM:SSZ"

}

- web_search と x_search を必ず使い、公式・信頼できるソースのみ参照

- 情報が見つからない場合は latest_ep_num を null にし、理由を備考に記載

- 出力はこのJSONオブジェクトのみ。余計なテキスト禁止。

このプロンプトなら、Grokが毎回「今何話まで出てるか」を正確に返してくれる可能性が高くなる。  
→ ep_num のインクリメントをこちらでやらなくて済む

局ごとの進捗を別軸で保持するメリット

- 放送局ごとに「最終放送話数」が違うことを自然に扱える
- ユーザー側で「MX派」「BS11派」など選択可能になる
- 「見てないのに次の話に進む」問題が根本的に解消

例：

- MX：第28話まで放送済み
- BS11：第27話まで  
    → MXユーザーには第29話の予告を、BS11ユーザーには第28話の情報を優先表示

次にやるべきこと（優先順）

1. Grokプロンプトを上記のように修正  
    → watch_list.json の ep_num が「Grokが確認した最新値」になるようにする
2. 局ごとの進捗を記録する仕組みを追加

- 新ファイル broadcast_history.json を作る（例：{ “anime_id”: { “mx”: 28, “bs11”: 27, … } }）
- daily_schedule.json 生成時に、局ごとの最新放送話を更新

4. ユーザー設定を想定した表示ロジック

- 将来的にユーザーごとの「視聴局リスト」を持つ（今は仮に全放送局を考慮でもOK）

これで「情報収集はGrok任せ」「局ごとの進捗は別管理」「ユーザーが見てない話に飛ばない」

はい、そういうケースが増えてるよね。Netflix独占先行配信 → 数ヶ月後に地上波順次放送、というパターンは2025〜2026年でかなり多くなってる（特にオリジナルアニメや海外共同制作作品）。

あなたの今の構造（watch_list.json + daily_schedule.json + 局ごとの進捗別管理）なら、十分対応可能だし、むしろ強みになる形にしやすいです。以下に具体的にどううまく使えるか、メリットと注意点をまとめるよ。

1. この構造がNetflix先行パターンに強い理由

- watch_list.json（監視対象リスト）  
    → 「Netflixで先行配信中」の作品も普通に登録できる  
    → ep_num を「Netflixで最新まで配信された話数」として更新  
    → 地上波開始前は「Netflix最新話 = 次に見るべき話」として扱える
- daily_schedule.json（今日の放送予定）  
    → 地上波が始まるまでは空（または「Netflix配信予定」として別フィールド追加可）  
    → 地上波開始後は「今日のMX/BS11/ABEMAの放送予定」だけが入る  
    → ユーザーが「Netflixで見てる」か「地上波で見てる」かを区別しやすい
- 局ごとの進捗別管理（broadcast_history.json など）  
    → Netflixは「配信局」として仮想的に扱える（例: station_id = “netflix”）  
    → 地上波は実局（mx, bs11, abema など）ごとに最終放送話数を記録  
    → 結果として「Netflixで見てる人は第8話まで進んでるけど、MXで見てる人はまだ第1話」みたいな差を正確に表現できる

2. 具体的な運用イメージ（Netflix先行の場合）

例：『SPY×FAMILY 新シーズン』（仮にNetflix独占先行 → 半年後に地上波）

- Netflix先行期間（2026年3月〜8月）

- watch_list.json  
    {
-   "title": "SPY×FAMILY 新シーズン",
-   "ep_num": 8,               // Grokが「Netflix最新話は第8話」と返した
-   "source_links": [{"type": "netflix", "url": "..."}],
-   "stations_last_ep": {"netflix": 8}
- }
-   
    
- daily_schedule.json  
    → 空（または「Netflix配信済み」としてフラグだけ立てる）

- 地上波開始後（2026年9月〜）

- daily_schedule.json  
    [
-   {
-     "anime_id": "...",
-     "title": "SPY×FAMILY 新シーズン",
-     "episode_number": 1,
-     "start_time": "2026-09-01T00:00:00+09:00",
-     "station_id": "mx",
-     "status": "normal"
-   }
- ]
-   
    
- stations_last_ep 更新  
    "stations_last_ep": {"netflix": 8, "mx": 1}
-   
    

- ユーザー体験

- Netflixで見てるユーザー → 「次は第9話（Netflixで配信待ち）」
- MXで見始めたユーザー → 「次は第2話（MXで放送予定）」 → 同じ作品でも「見てるプラットフォーム」によって次話が変わる

3. プロンプトをNetflix先行に対応させるコツ

Grokに聞くときのプロンプトを少し調整すると、さらに正確になる。

作品名：{{アニメタイトル}}

対象：この作品の**現在最も新しいエピソード**（配信・放送済みの最新話）

- Netflix先行配信の場合も含めて、最新の配信/放送済み話数を優先

- 地上波が始まっていない場合は「Netflix最新話」を基準に

出力JSONに以下を含めて：

{

  "latest_ep_num": 整数,

  "platform": "netflix" | "mx" | "abema" | "複数" など,

  "latest_broadcast_date": "YYYY-MM-DD",

  "is_netflix_exclusive_currently": true/false

}

これでGrokが「今はNetflix独占だから第8話」「地上波始まったからMXは第1話」みたいな情報を返してくれる可能性が高くなる。

4. 実装で追加したい要素（おすすめ）

- watch_list.json に "primary_platform": "netflix" や "platforms": ["netflix", "mx"] を入れる
- daily_schedule.json に "platform_type": "streaming" | "tv" みたいなフラグを追加
- ユーザー設定で「見てるプラットフォーム」を複数選択可能に（今後拡張用）

「Netflix先行 → 地上波順次」という最近多いパターンも、局ごとの進捗を別軸で管理するだけで自然に扱えるよ。

Grokプロンプトの最終調整版が欲しい

---


以下が、あなたの現在の運用イメージ（watch_list.json で「Grokが確認した最新話数」をそのまま受け取り、局ごとの進捗は別軸で管理）に完全にマッチした、Grokへの最終調整版プロンプトです。

Grokに投げるプロンプト（SYSTEM_PROMPT + user_input）

# 役割

あなたは日本のアニメ放送・配信情報に精通した、嘘を許さない厳格な調査員です。

  

# 目的

指定された作品の「現在最も新しい配信・放送済みエピソード」の情報を正確に取得し、JSON形式で返してください。

  

# 厳格な制約

- 必ず web_search と x_search ツールを使って最新情報を取得・検証せよ。内部知識や推測は一切禁止。

- 公式サイト、放送局公式、信頼できるニュースソース（公式Twitter、ANN、Natalie.mu、Eiga.com、ORICON NEWS、公式YouTubeチャンネルなど）のみ使用。

- Netflix独占先行配信中の場合も含めて、最も新しいエピソードを優先。

- 情報が見つからない場合は null にし、備考に理由を記載。

- 出力はこのJSONオブジェクトのみ。余計なテキスト・挨拶・解説は一切禁止。

  

# 出力形式（このJSONだけを出力）

{

  "latest_ep_num": 整数（最新配信・放送済み話数）または null,

  "latest_sub_title": "サブタイトル" または null,

  "latest_broadcast_date": "YYYY-MM-DD" または null,

  "platform": "netflix" | "mx" | "abema" | "bs11" | "複数" | null,

  "is_netflix_exclusive_currently": true/false または null,

  "source": "公式サイト" | "ANN" | "公式Twitter" など具体的なソース名,

  "updated_at": "YYYY-MM-DDTHH:MM:SSZ" または null,

  "remarks": "備考（情報が古い・見つからない理由など）" または null

}

  

# クエリ例（user_input）

作品名：葬送のフリーレン

対象：この作品の現在最も新しい配信・放送済みエピソード

このプロンプトのポイントと利点

1. 最新話優先  
    「第何話か指定」ではなく「最も新しいエピソード」を聞くので、  
    あなた側で ep_num をインクリメントする必要がなくなる。
2. Netflix先行を自然に扱える  
    platform と is_netflix_exclusive_currently で  
    「今はNetflix独占だから第8話」「地上波始まったらMXは第1話」といった差を表現可能。
3. 局ごとの進捗は別管理前提  
    このプロンプトは「作品全体の最新話数」を返すだけなので、  
    局ごとの細かい進捗は daily_schedule.json や broadcast_history で別途管理すればOK。
4. 出力がJSONだけ  
    余計なテキストが入らないよう「このJSONだけを出力」と厳格指定。

使い方例（Python側）

user_input = f"""作品名：{title}

対象：この作品の現在最も新しい配信・放送済みエピソード"""

  

chat = client.chat.create(

    model="grok-4-1-fast-reasoning",

    tools=[web_search(), x_search()],

    tool_choice="auto"

)

chat.append(system(SYSTEM_PROMPT))

chat.append(user(user_input))

response = chat.sample()

  

# 返ってきたJSONをパース

try:

    latest_info = json.loads(response.content)

    new_ep_num = latest_info.get("latest_ep_num")

    if new_ep_num is not None:

        anime["ep_num"] = new_ep_num  # watch_list.json に上書き

        logger.info(f"{title} の最新話数を更新: {new_ep_num}")

except:

    logger.error("JSONパース失敗")

これで、

- Grokが毎日「今何話まで出てるか」を教えてくれる
- あなたはそれをそのまま watch_list.json に反映
- 局ごとの細かい進捗は別ファイルで管理

という流れがスムーズに回るはずです。

Netflix先行パターンも自然にカバーできるので、最近多いケースにも対応できます。

もし「もう少し厳密にNetflixと地上波の区別を入れる」とか  
「出力JSONに追加したいフィールドがある」などがあれば、すぐに調整版作りますよ。

broadcast_history.jsonのサンプル構造

---



以下は、broadcast_history.json のサンプル構造と、その意図・使い方をまとめたものです。

目的

- 各作品ごとに、放送局（または配信プラットフォーム）ごとの最終放送/配信済み話数 を記録する
- 「Netflix先行 → 地上波順次」のようなケースでも、プラットフォームごとの進捗差を正確に保持
- daily_schedule.json や watch_list.json と連携して「ユーザーが見ている局の次話」を計算しやすくする

        "last_broadcast_date": "2026-03-15",

        "last_updated_at": "2026-03-20T23:45:00Z",

        "remarks": "Netflix独占先行配信中"

      },

      "mx": {

        "last_ep_num": 5,

        "last_broadcast_date": "2026-03-19",

        "remarks": null

      },

      "bs11": {

        "last_ep_num": 4,

        "last_broadcast_date": "2026-03-18",

        "last_updated_at": "2026-03-20T23:45:00Z",

        "remarks": "最速配信"

      }

    },

    "overall_latest_ep": 8

  },

  "anime_id_2": {

    "title": "呪術廻戦 第3期",

    "platforms": {

        "remarks": "地上波先行のため未配信"

      }

    },

    "overall_latest_ep": 15

  }

}

各フィールドの意味と役割

|   |   |   |   |
|---|---|---|---|
|フィールド名|型|必須？|説明・用途|
|[anime_id]|object|必須|作品ごとのキー（watch_list.json の anime_id と一致させる）|
|title|string|推奨|人間が読むためのタイトル（デバッグ・表示用）|
|platforms|object|必須|キー＝プラットフォームID（netflix, mx, bs11, abema, prime など）|
|└─ last_ep_num|integer|必須|そのプラットフォームで最終配信/放送済み話数（nullなら未配信）|
|└─ last_broadcast_date|string|推奨|最終配信/放送日（YYYY-MM-DD）|
|└─ last_updated_at|string|必須|この情報をGrokが最後に確認した日時（ISO8601）|
|└─ remarks|string|任意|備考（「Netflix独占中」「1話遅れ」など）|
|overall_latest_ep|integer|推奨|作品全体の最新話数（全プラットフォームのmax値）|

メリット

- Netflix先行の場合 → “netflix”: {last_ep_num: 8} が先に進む
- 地上波開始後 → “mx”: {last_ep_num: 1} などが追加されていく
- ユーザー設定（「MXで見てる」）と組み合わせれば、次回話数を正確に計算可能
- daily_schedule.json と連携 → 「今日MXで第6話やる → MXユーザーの次は第7話」

運用時のポイント

- 更新タイミング  
    daily_schedule.json 生成時に、放送実績を broadcast_history.json に反映（Grokが「今日放送された話数」を教えてくれる）
- 初期値  
    新規作品追加時は "platforms": {}（空オブジェクト）からスタート
- Grokプロンプトとの連携  
    最新話取得プロンプトに「プラットフォームごとの最新話数も可能な限り記載」と追加すると、さらに精度が上がる

この構造なら、Netflix先行・地上波順次のような複雑なケースも自然に扱えます。

はい、シーズン切り替わり時期のオーバーラップ（3月末〜4月初旬に旧シーズン最終話と新シーズン初回が重なる時期）は、確かに今の「シーズン毎にマルっと入れ替え」方式だと結構痛い問題になりますね。

今の仕組みだと、3月末に「2025年冬アニメ」のリストを丸ごとアーカイブして「2026年春アニメ」に置き換えると、以下のような問題が起きやすい：

- 3月最終週にまだ放送中の冬アニメの最終話が取れなくなる
- 4月1週目に始まる春アニメの第1話が、切り替え直後だと情報が薄い（公式更新が遅い場合）
- オーバーラップ週の「今日の放送予定」が中途半端になる（旧シーズン最終話と新シーズン初回が混在）

これを回避するために、「シーズン毎の完全入れ替え」ではなく「継続監視＋新旧混在」 にシフトするのが現実的です。

シーズン切り替わりをうまく扱うための構造変更案

1. watch_list.json を「シーズン縛り」から解放する

- 今のwatch_list.jsonは「そのシーズンの全作品」として丸ごと管理しているが、これを**「現在監視すべき作品のリスト」** に変える。
- シーズン終了後も「最終話まで監視したい作品」は残す（例: 人気作の最終回、未完結作）
- 新シーズン開始時は、既存リストに追加 するだけ（削除はしない）

2. シーズン情報を作品単位で持つ（watch_list.json にフィールド追加）

[

  {

    "title": "葬送のフリーレン 続編",

    "anime_id": "202601_frieren_s2",

    "season": "2026冬",

    "season_end_date": "2026-03-31",   // 新規追加：シーズン終了予定日

    "ep_num": 28,                       // Grokが確認した最新話数

    "is_active": true,                  // 監視継続中か（falseならアーカイブ済み扱い）

    "platforms": { ... }                // 局/プラットフォームごとの進捗

  },

  {

    "title": "新作春アニメA",

    "anime_id": "202604_newanimeA_c1",

    "season": "2026春",

    "season_end_date": "2026-06-30",

    "ep_num": 1,

    "is_active": true,

    ...

  }

]

- season_end_date を入れることで、3月末に「この作品はもう終わった」と判断しやすくなる
- is_active = false にしたらアーカイブフォルダに移動（または別リストに）

3. 情報収集のタイミングとロジック変更

- 毎日実行時

- watch_list.json の全作品（is_active=true）に対してGrokに「最新話」を聞く
- シーズン終了日を過ぎた作品は自動で is_active=false に（または警告ログ）

- 新シーズン開始時（4月1日など）

- 別途「2026年春アニメ」のリストを取得
- 既存の watch_list.json にマージ（重複チェックして追加）
- 旧シーズンの最終週は「season_end_date」を基準に監視継続

4. daily_schedule.json の扱い（オーバーラップ対応）

- 毎日「今日放送/配信される全ての予定」を集める
- watch_list.json の全アクティブ作品から、Grokに「今日の放送予定」を聞くように拡張
- 結果として、3月末〜4月初旬は旧シーズン最終話 + 新シーズン初回が混在しても自然に表示される

Grokプロンプトの微調整（シーズン切り替わり対応）

作品名：{{アニメタイトル}}

対象：この作品の現在最も新しい配信・放送済みエピソード

- シーズン終了間近の場合も、最新の配信/放送情報を優先

- Netflix先行配信中の場合も含めて、プラットフォームごとの状況を考慮

出力JSONに以下を含めて：

{

これで、シーズン切り替わり時期に「もうすぐ終わりそう」とGrokが教えてくれるので、

- watch_list.json を「現在監視中の作品リスト」に変える

おすすめの組み合わせ（最初はこれで始めるのがベスト）

- メイン：日付ベース（毎月25日頃に「次のシーズンの作品を10〜15件追加」）
- サブ：Grok自動検知（毎日1回「今から1ヶ月以内に始まる新作ある？」を聞いて、ヒットしたら追加）

Grokに「新作検知」を聞くための専用プロンプト例

watch_list.json に新作を追加するためのトリガープロンプト（別途Grokに投げる用）

現在の日付：{{today}}（YYYY-MM-DD）

対象期間：今から1ヶ月以内に日本で放送開始または配信開始する主要な深夜アニメを10〜15作品程度リストアップしてください。

  

出力形式（このJSON配列だけを出力）：

[

  {

    "title": "作品名",

    "season": "2026春" または "2026夏" など,

    "start_date_estimated": "YYYY-MM-DD" または null,

    "official_url": "https://..." または null,

    "is_netflix_exclusive": true/false または null,

    "expected_ep_first": 1,

    "source": "ANN" | "公式サイト" | "X公式" など

  }

]

  

- web_search と x_search を必ず使い、公式・信頼できるソースのみ参照

- 既にwatch_list.jsonに登録済みの作品は含めない

- 情報が不十分な作品は除外

これを毎月25日頃にGrokに投げて、返ってきたリストをwatch_list.jsonにマージ（重複チェック付き）すれば、新シーズンが自然に追加されていく。

全体の流れ（イメージ）

1. 毎日

- watch_list.json の全アクティブ作品について「最新話」をGrokに聞く → ep_num 更新
- daily_schedule.json を「今日の放送予定」として生成

3. 毎月25日頃（トリガー）

- 「今から1ヶ月以内の新作」をGrokに聞く
- 返ってきたリストをwatch_list.json に追加（重複チェック）
- 新規追加作品のmaster.json / broadcast_history.json 初期化

5. シーズン終了判定

- season_end_date を過ぎたら is_active = false に
- アーカイブフォルダへ移動

これで「シーズン切り替わり時期のオーバーラップ」も「新作の遅い発表」も両方カバーできるし、  
リストが突然消えたり増えたりする違和感もなくなるよ。

まとめとアドバイス

- 現状の構造では「最新話だけ上書き」なので過去情報が消え、局差も考慮できない
- 改善後では

- watch_list.json：監視対象＋最新話数（Grokが教えてくれる）
- broadcast_history.json：局ごとの最終進捗（これで「MXは第5話、Netflixは第8話」対応）
- episodes/：話数ごとの蓄積（過去あらすじも残る）
- daily_schedule.json：今日限定の放送予定（局ごとフィルタ可能）

この方向で進めると、Netflix先行・オーバーラップ時期も自然に扱えるし、  
「見てないのに次の話に進む」問題も解消できます。
