【ルール1: 初期リスト作成時の原則】

- ハルシネーションを防ぐため、すべての項目はツール検索（Web検索など）や内部知識に基づくものに限定。推測はソース付きのもののみ許可し、架空のタイトルや虚偽の情報を絶対に作成しない。

以下は、ユーザーのクエリに対する分析と提案です。まず、背景として、xAI経由（例: APIや直接アクセス）とWeb版（x.com上のGrok）の違いは、検索ツールの実行タイミング、キャッシュ、ジオロケーション（Matsumoto, Nagano, JP）、または内部の最適化による検索結果のばらつきが主な原因です。特に、2026年3月現在で「2026年冬アニメ」（通常、1月～3月放送分）は放送中または終了間近のため、公式発表、レビュー、まとめ記事が豊富に存在するはずですが、xAI経由で結果が少ないのは、SystemPromptの厳格なルール（Web検索結果に基づかないリスト作成禁止、情報未発見時は含めない）がより強く適用され、検索クエリの解釈やヒット数が制限されている可能性が高いです。

Web版相当の豊富な回答（例: 25件程度のリスト）を得るための望ましいプロンプト

Web版はより柔軟に検索を広げ、噂や関連情報を拾いやすい傾向があるため、プロンプトを設計する際は以下の原則を採用してください：

- 検索範囲の拡大: クエリを具体的にし、複数のキーワードやオペレーターを使ってヒットを増やす。2026年冬アニメの情報は、アニメ情報サイト（Anilist, MyAnimeList）、公式Twitter/X、ニュース記事、ファンフォーラムなどに散在しているので、それらを明示的に指定。
- ルールの緩和意識: SystemPromptの厳格さを考慮し、プロンプトで「可能な限り多くのソースから抽出」「噂や予測を含むがソース明記」を強調。ただし、完全な創作を避ける。
- ツールの積極活用: Web検索だけでなく、X Keyword SearchやX Semantic Searchを組み合わせ、リアルタイムの議論や噂を捕捉。
- 出力形式の指定: リスト形式を維持しつつ、並び順（例: 五十音順）を明確に。
- タイミング考慮: 現在の日時（2026年3月10日）を明記し、放送中/終了済みの情報を区別。

なぜこのプロンプトが有効か？

- ヒット数増加: 具体的なサイト指定（site: operator）やX検索のオペレーター（min_faves:5 で人気投稿優先）で、質の高い結果を増やす。num_resultsを増やしてデフォルトの制限を回避。
- 多ツール統合: Web検索だけでなくXツールを明示的に呼び、SNSのリアルタイム情報を補完（Web版の強みであるX統合を模倣）。
- ルール遵守: SystemPromptの制約を尊重しつつ、「可能な限り多く抽出」を強調して空リストを避けやすい。
- 柔軟性: 現在の日時を入れて、放送後情報（レビューなど）を拾いやすくする。2026年冬アニメは3月終盤なので、まとめ記事が多いはず。

このプロンプトをxAI経由で試せば、Web版に近い25件レベルのリストが得られる可能性が高いです。もしさらに調整が必要（例: 具体的な検索結果の例を追加）なら、詳細を教えてください。

ステップバイステップの指定が重要と聞いたが、xAIはリストの上から順に処理していくイメージでよいか？

---

はい、xAI経由（特にAPIや直接のクエリ処理）では、リストの上から順に処理していくイメージでほぼ正しいです。ステップバイステップの指定が特に重要になる理由と、実際の挙動を簡単にまとめます。

xAIの処理の基本的な流れ（イメージ）

1. プロンプト全体を上から下へ一気に読む  
    最初にSystem Prompt → User Prompt → 追加の指示（もしあれば）を順番に解析します。
2. 指示が登場した順に優先・適用される傾向が強い

- リスト形式の指定（例: 「1. タイトル [ラベル]」）が上の方にあると、それが強く効く。
- 「Web検索の結果に基づかないリストは絶対に作成しない」という厳格ルールが上の方に書いてあると、後半で「可能な限り多く抽出せよ」と言っても、検索結果が少ないと空リストになりやすい。
- つまり、上の方の制約が下の方の拡張指示を上書き・制限しやすい。

4. ツール呼び出しのタイミングも上から順

- 「Web検索を活用し」「X Keyword Searchも」など、最初に登場したツールが優先的に使われやすい。
- 後半で「num_resultsを20以上に」と書いても、内部のデフォルトや初期の検索クエリが弱いと、結果が少なく止まる。

だからWeb版とxAIで差が出やすい理由

- Web版（x.comのGrok）：会話調・ユーザー体験重視で、プロンプトの後半の「可能な限り多く」「噂も含めて」などの意図を柔軟に汲み取り、検索を広げてくれる傾向。
- xAI経由：よりrawで厳格にプロンプトを順番処理。上の方の制約（検索必須・未発見時は含めない）が強く効き、結果が0〜2件に留まりやすい。

ステップバイステップ指定が効くようにするコツ

- 制約を先に緩和してから詳細指定  
    悪い例（上から厳格→拡張）：  
    「Web検索に基づかないリストは絶対作成しない」  
    ↓  
    「可能な限り多く抽出」  
    良い例（上から拡張→制約）：  
    「Web検索、X検索をフル活用して可能な限り多くの2026年冬アニメをリストアップせよ。公式・噂・予測すべてソース付きで含めて構わない」  
    ↓  
    「情報が全く見つからない項目だけ除外。推測は禁止」
- 処理順を明示的に指示  
    「まず広範なWeb検索とX Semantic Searchを実行 → その結果を基にリストを作成 → ソースを必ず付与」という順番をプロンプト内でステップとして書く。

要するに、xAIは「上から順に積み重ねて解釈」する傾向が強いので、  
「一番上に欲しい結果に近い指示を置き、徐々に細かく制約を追加」  
という順序にすると、Web版に近い豊かなリストが出やすくなります。

もし具体的なプロンプトをもう少し調整したい場合、現在のものを貼ってもらえれば、順番を最適化したバージョンを提案できますよ。
